- 会場一致かつ指定R不一致は未発売/締切であることが多いため、入力失敗として落とすより `unavailable_race` で見送ると運用の連続性を保ちやすい。

## 2026-02-19: 競輪TOP一覧は会場優先フォールバックが有効

## 2026-03-01: トップ復帰判定はURL/一覧要素で厳密化する
- `#loginbox2` のような共通領域だけでトップ復帰完了を判定すると誤検知するため、`/pc/top` URL または開催一覧要素の表示を併用する。

- トップ到達直後でも確認ダイアログが後追い表示されるため、復帰完了前に「短い待機 + OK再確認」を入れると取りこぼしを減らせる。

## 2026-03-01: 詳細デバッグログは解決後に整理する
- 原因切り分けのために段階ログを一時追加した場合、再現解消後は失敗時ログだけ残して通常運用ログを簡潔に戻すと監視しやすい。

## 2026-03-02: 固定スケジュールはUIで時刻変更できる形が運用しやすい
- 監視常駐運用では「固定スケジュールON/OFF + 開始/終了時刻」を設定画面で変更可能にすると、季節や開催時間変更への追従が速い。

## 2026-03-02: 設定共有用JSONは秘匿項目だけ空にした雛形が安全
- 同一運用設定を使い回す場合は、`ID/パスワード/認証ID` のみ空にしたテンプレートを用意すると、再設定が速く情報漏えいも防ぎやすい。

## 2026-03-02: 競輪TOP一覧は背景色クラス固定で絞らない
- 開催一覧の行は運用タイミングで背景色クラスが変わるため、`li.kyotuHeader.white_bg_color` の固定セレクタでは会場行を取りこぼす場合がある。`li.kyotuHeader` ベースで探索する方が安定する。

## 2026-03-02: 事前候補通知は受信種別で実行経路を分離する
- 連携メッセージを1種類で扱うと誤発注リスクがあるため、`message_type` で `pre_race_candidates` を明示して投票キューから分離するのが安全。
- 候補通知は「ログ反映のみ・発注なし」をテストで固定化しておくと、将来のパーサ拡張時にも回帰を防ぎやすい。

## 2026-03-02: 再起動耐性は「履歴再取得」を候補通知だけに限定する
- 起動時にDiscord履歴を再読込する場合、発注JSONまで対象にすると重複実行リスクがあるため、`pre_race_candidates` のみ再取得する設計が安全。
- 再取得は当日データに絞ることで、過去日のノイズを避けつつ「8時配信を起動後に補足する」運用要件を満たしやすい。

## 2026-03-04: 複数戦略通知を受ける受信側は「戦略フィルタ」を明示実装する
- 同一チャンネルで複数戦略を流す場合、受信側に `dispatch_mode`（勝負/平均/守り/全部）を持たせると誤発注を防ぎやすい。
- 既存通知との互換のため、`strategy_code` が無い旧payloadは受け入れる挙動を維持するのが安全。

## 2026-03-04: 事前候補は「暫定通知」の注記を受信側でも見える化する
- `pre_race_candidates` は最終買い目ではないため、`is_provisional/advisory` を受信ログへ残すと、直前で見送りに変わる挙動を運用者が誤解しにくい。
- 候補通知は発注キュー非投入のまま、注記ログだけ追加する形が安全に運用改善できる。

## 2026-03-04: 戦略フィルタは「全許可モード」を持たない方が事故を減らせる
- 受信戦略に `全部` を残すと、モード分離運用でも意図せず全エントリーが通るため、`勝負/平均/守り` の明示選択のみ許可する方が安全。
- 既存設定互換は `dispatch_mode=all -> average` の自動移行で吸収すると、UI簡素化と後方互換を両立しやすい。

## 2026-03-08: 将来拡張の戦略コードは早めに共通正規化へ寄せる
- `config / parser / service / UI` に同じ戦略名変換を重複実装すると、6モード化のたびに差分がずれやすい。共通 helper に寄せる方が安全。
- 戦略コードの命名が後で変わる可能性がある場合でも、alias 吸収を共通層へ集約しておけば受信側の互換維持がしやすい。

## 2026-03-09: GUIで隠した設定はロード時にも固定化する
- 項目を画面から外すだけでは旧設定JSONの値が残って実動作が変わるため、`config` 側で hidden setting を既定値へ揃える方が安全。
- 接続デバッグやkeepaliveのような内部ログは、サービス出力は残したまま UI 側で非表示にすると、切り分け用ログとお客様向け表示を両立しやすい。

## 2026-03-09: 「固定」の意味は金額固定か払戻固定かを先に確定する
- 投票UIの「固定」は案件によって `1点固定金額` と `目標払戻固定` の両義があるため、表記変更タスクでは先にどちらを残すか確認してから UI と既定値を揃える方が安全。

## 2026-03-15: 送信側の戦略整理後は受信側のフィルタを先に疑う
- 通知パターンが実運用で1本化されたら、受信側の戦略フィルタは価値より事故要因になりやすい。UIとサービスをセットで外す方が安全。
- 新しい `strategy_code` が入り続ける連携では、未知コードを拒否せずログ保持だけにすると運用停止を避けやすい。

## 2026-03-15: 候補なしを見せたいなら空候補 payload を受け入れる
- 候補通知を「有るときだけ送る」設計にすると、受信側では `候補なし` を表現できない。運用上の見える化が必要なら、`candidate_count=0 / candidates=[]` の payload をそのまま通す方が扱いやすい。
- `pre_race_candidates` parser は `candidates=[]` を機械的に弾かず、サービス層で `候補なし` と表示した方が、表示文言変更とキュー制御を分離しやすい。

## 2026-03-15: 候補通知でブラウザ制御するなら配信時刻を条件に含める
- `候補なし` や `ミッド候補あり` だけでブラウザ開閉すると、手動再送やテスト通知でも実ブラウザが動いてしまう。`requested_at` の 08時/18時 と `filters.exclude_midnight / midnight_only` を組み合わせて制御すると安全。
- 固定スケジュール起動と候補連動停止を併用する場合は、「当日だけ有効な pause フラグ」を持たせて、後続の自動起動が即座に復活しないようにすると扱いやすい。

## 2026-03-16: Discord受信コールバックでは同期Playwrightを直接触らない
- `discord.py` の `on_message` コールバックは `asyncio` ループ上で動くため、候補通知を受けた直後のブラウザ制御でも同期版 Playwright を直接呼ぶと `Sync API inside the asyncio loop` 例外になる。
- 受信スレッドではログ記録とキュー投入だけに留め、Playwright を使う停止/再起動/発注は既存ワーカースレッドへ寄せる方が安全。

## 2026-03-18: プラン別受信制御は「枠数固定」でUIに落とすと迷いにくい
- `BRONZE=1枠 / SILVER=2枠 / GOLD=全枠` を自由選択式にすると、保存時の補正ルールが見えづらい。クリックで入れ替える固定枠UIの方が商品版では説明しやすい。
- Discord候補通知を時間帯別に制御する場合は、`strategy_code` だけでなく `requested_at` の時刻も併用すると、08時/10時/18時の自動停止・再起動条件を安全に分けやすい。

## 2026-04-05: Playwright の Temp 異常は起動前に吸収する
- Windows実機では `%TEMP%` が壊れていて `mkdtemp(...playwright-artifacts-XXXXXX)` が失敗することがあるため、Playwright 起動前に書き込みテスト付きで代替 Temp を選び直す方が安全。
- `sync_playwright()` を長時間運用コードへ組み込む場合は、サービス側の呼び出し元スレッドに依存させず、専用オーナースレッドへ固定した方が `asyncio` 競合を避けやすい。

## 2026-04-05: 同梱版Playwrightは `chromium` だけでは不足する
- `headless=True` の既定運用では Playwright が `chromium_headless_shell-*` を参照するため、PyInstaller 同梱対象を `chromium-*` だけにすると `chrome-headless-shell.exe` 不足で壊れる。
- 配布版は spec 側で `chromium_headless_shell-*` も拾い、実行側でも欠落時に `chromium-*` 実行ファイルへ逃がす二段構えにすると事故を減らせる。

## 2026-04-05: Windowsの既定ブラウザ設定は Playwright の起動先決定に使われない
- 配布先PCに Chrome が入っていても、`browser_channel` が空欄だと自動で Chrome を選ぶとは限らない。Windowsの「既定のブラウザ」と Playwright の起動チャンネルは別物として扱う方が安全。
- 軽量EXEで運用するなら、`browser_channel` 明示か、未指定時に `Chrome -> Edge -> Playwright Chromium` の順で自動試行する実装が実用的。

## 2026-03-18: 商品配布でプランを本当に分けるならEXE側にも固定値を持たせる
- UIでプラン変更を隠すだけでは設定JSONから戻せるため、配布版はビルド時に固定プラン値を埋め込み、`config` の読込時にも上書きする方が安全。
- PyInstaller の runtime hook で固定プラン環境変数を注入すると、コード分岐を最小限に保ったまま `BRONZE / SILVER / GOLD` の別EXEを作り分けやすい。

## 2026-03-18: 固定版の現在値表示と「選択肢一覧表示」は分けて考える
- 固定プラン前提の helper をカード一覧描画にも流用すると、`BRONZE / SILVER / GOLD` の一覧がすべて固定値表示になってしまう。現在値用 helper と一覧用 helper は分けた方が安全。
- 2枠選択UIで毎回ソートしてしまうと、クリック順に依存する入れ替えが成立しない。SILVER のような固定枠UIは、内部では選択順を保持してから表示時だけ整列する方が扱いやすい。

## 2026-03-18: 旧UIを残す配布物は「別ビュー + 実行時フラグ」で切ると安全
- 商品版UIと体験版UIを同じ画面クラスで条件分岐し始めると崩れやすい。旧UIを別Viewとして残し、起動時の variant フラグで切り替える方が追いやすい。
- 体験版のように `旧UI + GOLD固定` を配布する場合は、UI種別と固定プラン値を runtime hook で同時注入すると、保存設定やビルド引数との整合を保ちやすい。

## 2026-03-19: 上限選択UIでは「未設定」と「空設定」を分けて扱う
- 利用枠を「最大数まで選べる」仕様にすると、設定キーが未保存なのか、ユーザーが意図的に空配列 `[]` を保存したのかで意味が変わる。`config` 読込時はキー有無を見て既定値補完と明示空設定の維持を分ける方が安全。
- 上限式の選択UIは、上限到達中に別枠を押して自動入れ替えすると意図が読み取りにくい。先に解除してから追加するトグル挙動の方が「最大数まで」の説明と一致しやすい。

## 2026-03-25: 起動時の候補再取得と自動起動は同時に走らせない
- 起動直後に `pre_race_candidates` の履歴再取得を行う構成では、待機スケジュール自動起動を同時に動かすと候補連動の再起動と競合しやすい。再取得が終わるまで自動起動を待たせる方が安定する。
- ブラウザ起動失敗の画面ログは「失敗した事実」だけでなく例外詳細も出すと、環境差によるログインフォーム未検出や状態確認失敗の切り分けが速い。

## 2026-03-26: 固定版EXEだけ別の運用導線を持たせるなら環境フラグで切る
- `BRONZE / SILVER / GOLD` 固定版だけ手動確認を入れたい場合は、保存設定へ露出せず `fixed_plan` 環境フラグ起点で review callback を切ると、汎用EXEや `trial` への影響を最小化しやすい。
- 既存のレビュー処理が残っている案件では、UIから消していても service 側の review hook を再利用すると差分を小さく保てる。

## 2026-03-26: 固定版の実行方法切替は「既定値」と「保存値維持」を分ける
- 固定版だけ `自動 / 手動` を選ばせたい場合、`config` 読込時は未保存なら `手動` を既定にしつつ、保存済み `confirm_each_race=false` は上書きしない方が運用しやすい。
- review callback は固定版で常に渡しておき、実際に確認ダイアログを出すかどうかは `confirm_each_race` で決めると実装差分を小さく保てる。

## 2026-03-27: 上限1枠のUIは「別枠クリックで即置換」が自然
- `BRONZE` のように選択上限が1件しかない場合は、上限到達中でも別枠クリックを無視するより、その場で新しい枠へ置き換える方が操作感に合いやすい。
- 上限到達時の挙動はプラン数ごとに分け、`BRONZE` だけ置換、`SILVER/GOLD` は既存のトグル挙動維持にすると影響を最小化しやすい。

## 2026-03-30: 候補なし通知では実ブラウザを閉じない方が安全
- 候補通知を「当日の自動起動停止」の意味で使う場合、既存の `page/context/browser` まで閉じると、その後の発注処理が古い参照を握って落ちやすい。
- 候補なし時は pause フラグだけ立て、実ブラウザの破棄は終了時刻や明示再起動に任せる方が運用が安定する。

## 2026-04-04: 候補連動の件数判定は `candidate_count` を正にする
- `pre_race_candidates` の件数判定を `len(candidates)` に寄せると、送信側が0件通知を明示していても配列残りで再起動側へ誤判定し得る。
- ブラウザ制御や `候補なし` 表示は `candidate_count` を優先し、配列は候補プレビュー表示用として扱う方が安全。

## 2026-04-04: Windows既定ブラウザは Chrome 前提にしない
- 配布先のPCごとに Chrome / Edge / Playwright Chromium の入り方が異なるため、既定値で `browser_channel=chrome` を持つと起動事故になりやすい。
- 既定は Playwright Chromium に寄せ、特定ブラウザ指定は任意設定にしつつ、失敗時も Chromium へフォールバックする方が運用しやすい。

## 2026-04-10: 競艇通知の受信互換はプレーンテキストも先に吸収する

## 2026-04-10: 接続設定を複数ビューに持つ場合は全ビュー同時更新が必要
- 設定専用画面だけを競艇用に直しても、実際に表示される `dashboard` や `classic dashboard` に旧項目が残っていると、必須項目の `login_id` が保存されず自動開始が止まる。
- 認証項目の名称変更では、表示ラベルだけでなく `load_from_settings` と `collect_to_settings` の対応キーまで同時確認する方が安全。

## 2026-04-10: 競技固定の自動投票は候補 payload 側でも競技一致を確認する
- 同じ Discord チャンネルや履歴再取得を共有する運用では、`tool_slot` や `candidate_count` だけで候補を受けると、別競技の payload を拾って誤起動しやすい。`venue`、`provider`、`source`、`market` から競技一致を先に判定する方が安全。
- 競艇のように対象競技が明確なアプリでは、`unknown` を通すより `競技判定不可` として見送る方が、誤ログイン防止の優先度が高い。

## 2026-04-10: 候補連動のブラウザ制御は payload 単位ではなく日付と時間帯で重複排除する
- `pre_race_candidates` が分割通知や履歴再取得で複数回来る運用では、payload ごとに restart/pause を実行すると同じ時間帯に何度もログインし直す。
- 候補連動の action は `date + tool_slot + action` 単位で一度だけ実行し、次の時間帯や別日に入ったら自然に再許可する設計が安定する。

## 2026-04-10: コピー元案件から作った自動投票ではチャンネルID固定値の置換漏れを先に潰す
- `service.py` の受信判定を直しても、`secrets_local.py` にコピー元の Discord チャンネルIDが残っていると、起動時履歴再取得が別案件の履歴を読み続ける。
- 新規派生案件では、Bot token だけでなく `DISCORD_CENTRAL_CHANNEL_IDS` と `DISCORD_LOCAL_CHANNEL_IDS` の実運用値確認を初回確認項目へ入れる方が安全。

## 2026-04-10: Playwright の `fill()` だけで通らない入力欄は複数手段を持つ方が安全
- 公式投票サイトの確認画面では、要素が見えていても `fill()` が失敗することがある。クリック、`Ctrl+A`、`Backspace`、`type()`、最後に JS 代入まで順に試すと通りやすい。
- 特にパスワード欄は DOM 上の `id` が合っていても操作制御で落ちることがあるため、「セレクタ不一致」と「入力不能」は切り分けてログを見る方がよい。

## 2026-04-10: 自動入力失敗は「要素なし」と「要素ありだが入力不能」を分けて出す
- 同じ `selector missing or not fillable` では、DOM 更新漏れなのか、サイト側の入力制御なのかを即座に判別できない。`selector missing` と `input not fillable` を分けると運用切り分けが速い。
- 実機ログに CSS セレクタも残すと、HTML 断片を受け取った時点でコード修正箇所をほぼ特定できる。

## 2026-04-10: 確認画面の入力欄は `visible` 判定だけに依存しない
- 公式サイトでは確認画面の入力欄が DOM 上に存在しても Playwright の `is_visible()` で拾えないことがある。入力対象は「可視優先、だめなら attached 要素」まで見る方が実運用に強い。
- クリック系は可視要素を優先しつつ、入力系だけ探索条件を緩めると誤操作を増やさずに通過率を上げやすい。

## 2026-04-10: Playwright locator が空振りしても HTML 上の要素存在を否定しない
- 実機では `page.content()` に `#pass` が出ているのに locator 探索だけ失敗するケースがある。確認画面のような単純フォームは `document.querySelector(...)` の最終フォールバックを持つと詰まりにくい。
- とくに障害報告で HTML 断片が来た場合は、「その HTML に要素があるか」を最優先事実として扱い、locator 側の不安定さを疑う方が修正が早い。

## 2026-04-10: 配布版の切り分けには起動ログへ版番号を出す
- WSL のソースを直しても、Windows 側で古い EXE を起動していると症状が変わらない。起動直後に版番号を出しておくと「未反映」か「修正しても未解決」かをすぐ切り分けられる。
- 配布EXEはファイル名に版番号を含め、実行時ログでも同じ版番号を出すと運用確認が速い。

## 2026-04-10: 自動入力の障害解析は HTML 全体だけでなく selector probe を残す
- 全HTMLとスクリーンショットだけでも追えるが、毎回 `#pass がある/ない` を人手で探すのは遅い。selector ごとの `locator_count / query_selector_found / DOM抜粋` を JSON で残すと切り分けが速い。
- Playwright の locator と `document.querySelector` の結果が食い違うケースでは、その差分自体を保存すると修正方向が明確になる。

## 2026-05-03: 対応済み券種は実行側と判定側の両方へ明示する
- 公式サイト投入側で券種コードが定義済みでも、受信時の競技判定リストに同じ券種が無いと将来の通知形式変更で見送りになり得る。三連複のような共通名の券種は、会場や provider 判定と合わせてテストで固定すると安全。

## 2026-05-11: 公式サービス時間外はブラウザ常駐より明示停止を優先する
- BOAT RACE 公式サイトは時間外にサービス時間外ページを返すため、早朝からログイン画面を開きっぱなしにしない方が安定する。
- 起動スケジュールだけでなく、事前準備、候補連動再起動、発注 payload の各入口で時間外起動を抑止すると、メモリ解放と誤状態回避を両立しやすい。

- `ERR_CERT_AUTHORITY_INVALID` は `page.goto()` の時点で落ちるため、同じ page で粘らず `ignore_https_errors=True` の context を作り直して再試行すると候補連動を止めずに復旧しやすい。
